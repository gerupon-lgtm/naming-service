## 2. はじめに（背景・目的）

- **背景**: 姓名判断エンジンを実装したアプリを作りたいと思いついたことがきっかけ。診断ロジックを逆利用すれば「良い診断結果になる名前」を提案できるのではという発想から、ペットの命名支援機能に発展した。
- **目的**: 姓名診断（既存の姓名を判定する）機能と、ペット（犬・猫・小動物）向けの命名候補提案機能を持つ無料のWebサービスを、個人開発・低コストで公開する。
- **期待効果**: 【想定】ポートフォリオ・個人開発の実績として活用できる。姓名診断＋ペット命名という組み合わせはSNSでの話題化・拡散も期待できる。

## 3. スコープ（対象範囲・対象外）

### 対象範囲

- 姓名診断機能: ユーザーが既存の姓名（人間）を入力し、画数ベースのロジックで診断結果を表示する
- ペット命名提案機能: 犬・猫・小動物（うさぎ・ハムスター等）数種を対象に、苗字を含まない名前候補を提案する
- 提案時のオプション指定: 使いたい文字（must-include）、希望よみがな（逆引き生成）、出力文字種（漢字／ひらがな／カタカナ／アルファベット／なんでもいい）、対象動物、性別、希望カテゴリ

### 対象外・保留項目

| 対象外・保留項目 | 理由 | 再検討条件 |
|------------------|------|------------|
| 人間の赤ちゃん命名（新規名の提案） | 苗字を含めた五格計算との両立が必要になり規模が大きくなるため | 今回は完全に対象外（構想段階で決定済み） |
| アカウント・ログイン機能 | MVPをシンプルに保つため | 将来的に検討の可能性あり |
| 診断結果のDB保存（履歴機能） | 無料サービスでアカウント管理を増やさない方針 | URLパラメータ方式でのステートレス共有に加え、結果ファイル出力・クリップボードコピー（F-009, F-010）で代替 |
| 読み用マスタ（character_reading等）の新設 | kanjiapi.devのreading逆引きAPIで代替可能なため | API側の仕様変更・廃止時に再検討 |
| kanjiapi.dev障害時の実行時フォールバックAPI（jisho.org等） | 各APIの仕様・信頼性が未確認、実装コストに見合わない | kanjiapi.devの障害が実運用で頻発した場合に再検討（※Kanji alive／MuzukanjiAPIは「ビルド時の部首補正」用途では採用済み。ここで対象外なのは実行時のフォールバックとしての利用のみ） |
| 独自の名乗り読み（キラキラネーム的な読み）への対応 | 発生してからでないと定義できないため | 対象外（恒久） |

## 4. 用語定義

| 用語 | 定義 |
|------|------|
| 姓名判断 | 姓名の画数等から運勢・傾向を判定する伝統的な占い手法 |
| 五格（天格・人格・地格・外格・総格） | 姓名判断で使われる画数の集計区分。天格・人格・外格は姓を含む計算のため、苗字なしのペット候補では地格・総格のみを使用する |
| 熊崎式 | 姓名判断における代表的な流派の一つ。日本で最も普及している。部首によっては新字体の画数ではなく伝統的な部首画数（旧字体基準）を用いる独自の補正ルールを持つ（例: 氵→「水」の4画、扌→「手」の4画で数える） |
| 名乗り読み | 人名にのみ用いられる、辞書の通常の音訓読みとは異なる読み方 |
| モーラ | 日本語の音の単位（拍）。拗音・長音・促音は1モーラとして扱う |
| must-include | 「使いたい文字」機能における、結果に必ず含める文字の指定方式 |

## 5. 業務要件

- 姓名診断: ユーザーが姓を含む姓名を入力すると、都度 `character_master` の画数データを用いて五格を計算し、診断結果（総合点・ランク）を表示する。結果はDBに保存せず、URLパラメータに条件を埋め込むことで再計算・共有を可能にする。あわせて、ロジックで確定した診断結果をLLMに渡し、解説・肉付けコメントを生成する（F-012）。
- ペット命名提案: ユーザーが対象動物・性別・希望カテゴリ・使いたい文字・希望よみがな・出力文字種を指定すると、`name_master` に登録された候補（または読み逆引きで動的生成した候補）から、条件に応じた合成スコア（性別の重みを特に重視）でランキングした候補を提示する。

### 6.2 機能詳細（主要機能のみ）

**F-001 姓名診断**
- 入力: 姓・名（漢字／かな等）
- 処理: 入力文字を1文字ずつ `character_master` で画数変換し、五格（天格・人格・地格・外格・総格）を計算。`character_master` に文字がない場合は kanjiapi.dev に問い合わせ、取得結果を `character_master` にキャッシュする
- 出力: 総合点・ランク（SS〜C）・五格の内訳（F-012のLLMコメントを合わせて表示）
- **【確定・v1.8】流派**: 熊崎式を採用する。将来的に他の流派も選べるようにする方針のため、計算ロジックは流派を差し替え可能な構造（プラガブル）で実装する。ただしPhase1では熊崎式のみをハードコードし、複数流派管理用の `fortune_method` テーブルはまだ作らない（データ構造設計10章参照）
- **【確定・v1.7】スコア→ランク初期閾値**: SS=90以上／S=80〜89／A=65〜79／B=50〜64／C=49以下（実装中に調整可能な設定値として持つ）
- 例外: kanjiapi.devにも該当文字がない場合、診断不可の旨をエラー表示する。**【確定・v1.7】文言**:「一部の文字の画数情報を取得できませんでした。表記を変えて試すか、しばらくしてから再度お試しください。」

**F-012 姓名診断結果へのLLMコメント生成**
- 入力: F-001で計算済みの構造化データ（五格・スコア・ランク）
- 処理: ロジックで確定した診断結果をそのままLLMに渡し、結果の意味・傾向を解説する自然文コメントを生成する（ロジック側で結果を確定させ、LLMは「肉付け」役に徹する。診断結果そのものをLLMに計算させない）
- 提供元: F-011と同じくOllama（自宅サーバー）を第一候補、OpenRouterへフォールバック。フォールバック順・モデルも同じ環境変数で共用管理する
- 表示方法: 診断結果本体は即時表示し、LLMコメントは非同期で後から表示する（コメント生成に時間がかかっても、五格・スコア・ランクの表示は待たせない）【想定】
- 失敗時: Ollama・OpenRouterとも応答不可の場合、コメントなしで診断結果のみ表示する（F-011と同じフォールバック方針）

**F-002 ペット命名候補提案**
- 入力: 対象動物（犬／猫／小動物）、性別、希望カテゴリ、使いたい文字（任意・複数）、希望よみがな（任意）、出力文字種
- 処理: `name_master` の候補、または読み逆引きで動的生成した候補に対し、地格・総格スコア、響き、カテゴリ、性別（重み大）を合成スコアで評価しランキングする。使いたい文字と出力文字種が矛盾する場合はエラー表示する
- 出力: スコア順の候補名リスト（各候補にF-011のLLMコメントを付与）

**F-009 結果ファイル出力**
- 対象: F-001（姓名診断結果）、F-002（命名候補リスト）
- 処理: 表示中の結果をテキスト（またはCSV等）としてブラウザ経由でダウンロードする。サーバー側にファイルを保存する必要はなく、クライアント側で生成する
- **【確定・v1.7】出力形式**: テキスト（.txt）のみとする。CSV／JSON等の対応は将来必要になれば追加する

**F-010 結果クリップボードコピー**
- 対象: F-001、F-002
- 処理: 表示中の結果（診断内容または候補名リスト）を1クリックでクリップボードにコピーする。ブラウザのClipboard APIを利用する想定

**F-011 候補名へのLLMコメント生成**
- 対象: F-002で表示される命名候補すべて
- 処理: 候補名ごとに、由来・意味も含めたやや長めの説明文をLLMで生成する
- 提供元: **Ollama（自宅サーバーを外部公開・接続実績あり）を第一候補とし、応答不可時はOpenRouterの無料枠モデルにフォールバックする**
- **フォールバック機能・フォールバック順の指定は、MVP時点から実装する**（後続フェーズへの先送りはしない）
- フォールバック順は設定で指定可能にする（デフォルトは `ollama → openrouter`。将来的に順序を入れ替えたり、片方のみを使う設定も可能にする）
- **使用モデルも設定で変更可能にする。** 初期想定モデルは以下の通り（両プロバイダともgemma4系で統一）
  - Ollama: `gemma4:31b-cloud`（Ollama Cloud経由）
  - OpenRouter: `google/gemma-4-31b-it:free`
- 各プロバイダともに無料利用のため、コスト・呼び出し回数は考慮不要とする
- **【想定】応答不可の判定基準**: タイムアウト5秒、またはHTTP 5xx／接続エラーを「応答不可」と判定し、次の候補（設定順の2番目）へフォールバックする。全プロバイダが応答不可の場合はコメントなしで候補名のみ表示する（リスク欄に対応済み）
- **【想定】設定方法**: 環境変数で管理する（個人開発・Vercel環境変数機能との親和性を優先し、設定画面は作らない）。例:
  - `LLM_FALLBACK_ORDER=ollama,openrouter`（カンマ区切りで順序を指定。`openrouter,ollama` や `ollama` 単独も可）
  - `LLM_OLLAMA_MODEL=gemma4:31b-cloud`
  - `LLM_OPENROUTER_MODEL=google/gemma-4-31b-it:free`

## 7. 非機能要件

| 観点 | 要件 |
|------|------|
| 性能 | 同時利用者数は小規模（個人サービス規模）を想定。画面表示は「診断中」等のプログレス表示を出せば10秒程度まで許容 |
| 可用性 | 【想定】個人運営のためSLAは設定しない。ベストエフォートでの提供とする |
| セキュリティ | ログイン機能なし、個人情報は収集しない。診断結果もDBに保存せずURLパラメータで共有する方式のため、個人情報保護上の懸念は小さい。将来アカウント機能を追加する場合は認証方式を別途検討する【要確認】 |
| 運用・保守 | 開発・運用は開発者本人のみが担当する。障害対応・問い合わせ対応も本人対応とする |
| 移植性・互換性 | スマートフォンを含むレスポンシブ対応とする（確定）。対象ブラウザはレンダリングエンジンChromium系（Chrome/Edge等）とWebKit系（Safari）の2系統を主対象とする |

## 8. データ要件

- データ構造の詳細は別紙「命名サービス向けデータ構造設計.md」（v11）を参照する。本書ではその要点のみ記載する。
- 提案用候補データ（`name_master`）は、各対象動物種あたり数百件程度の初期データ規模を想定する
- `character_master` は一般的な文字（常用漢字等）を網羅する方針とし、未収録の文字は kanjiapi.dev からの取得結果をキャッシュして拡充する
- 姓名診断（F-001）の結果はDBに保存しない。ペット命名候補（`name_master`, `fortune_result`, `fortune_detail`）のみ永続データとして保持する

## 11. 制約条件・前提条件

- **費用**: 初期費用・運用費ともに基本0円を目標とする（無料枠のAPI・ホスティングのみ利用）

- **前提**: LLMはOllama（自宅サーバー、外部公開・接続実績あり）を第一候補とし、応答不可時はOpenRouterの無料枠モデルにフォールバックする。フォールバック順は設定可能とする

## 13. リスクと課題

| リスク/課題 | 影響 | 対応方針 |
|-------------|------|----------|
| kanjiapi.devの障害・レート制限 | 姓名診断・よみがな逆引き提案が一時的に利用不可になる | MVPではフォールバックAPIを実装せず、エラー表示のみで対応。実運用で障害が頻発する場合にフォールバック実装を再検討する |
| 独自の名乗り読みが未収録 | 一部の読み・文字で提案候補が出ない場合がある | 対象外として割り切り済み。診断不可時のメッセージ設計のみ検討する |
| 開発者一人での運用 | 多忙時・不在時に障害対応や保守が滞る可能性 | 個人開発の趣味プロジェクトとして許容する【想定】 |
| 自宅サーバー（Ollama）の可用性 | 自宅回線・サーバーの障害、停電、メンテナンス等でOllamaに接続できない期間が発生し得る | OpenRouterへのフォールバックで基本的に吸収する。両方とも応答不可の場合はコメントなしで候補名のみ表示する【想定】 |
| LLM出力の品質 | 説明文の内容が不自然、または事実と異なる由来を生成する可能性がある | 【想定】あくまで補助的な一言コメントとして扱い、正確性の保証はしない旨をUI上に軽く明記する |
| 熊崎式の部首補正とkanjiapi.devの画数不一致 | kanjiapi.devは伝統的な部首補正（氵→水4画等）を反映しない現代画数のため、未知文字を自動キャッシュした場合に熊崎式の正しい画数と異なる可能性がある | 補正が必要な代表的な部首・文字はシードデータ側で手動補正して`character_master`に投入する。Kanji alive API（第一候補）とMuzukanjiAPI（第二候補、Kanji alive対応外を埋める）の部首情報を使って対象文字の検出を半自動化する（データ構造設計v11参照）。それでも自動キャッシュされた未知文字は補正対象外となる限界を許容する【想定】 |
