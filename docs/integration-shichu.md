## 1. 目的・背景

### 1.1 やりたいこと【Q1で確定】
**姓名診断（メイン機能）**において、生年月日の入力があれば四柱推命的に「不足している五行」を求め、**診断中の名前がその五行を補えているかを参考情報として表示する**。姓名判断（熊崎式）だけでは扱えない「生まれ持った五行バランス」を、診断の読み解きに加える。

### 1.2 適用範囲【Q1で確定】

| モード | 生年月日突合 | 理由 |
|--------|--------------|------|
| **姓名診断（S-001/S-002）** | **使う** | 本人の名前を診断する場面であり、生年月日が分かっている前提が成り立つ |
| **ペット命名（S-003/S-004）** | **使わない** | ペットの生年月日は不明なことが多い。必要なユーザーは診断モードで実行すればよい |

> この判断により、**ペット命名の提案ロジック（`api/_lib/pet/suggest.ts`）は本連携で変更しない**。既存の複合スコア（性別0.40／画数0.35／カテゴリ0.25）はそのまま。

### 1.3 なぜ四柱推命が連携相手として適任か
両エンジンのソースを確認した結果:

- **四柱推命エンジン（diagnose.js）は、そのまま使える五行バランスを既に出力している。**
  - `meishiki.gogyoCount` = `{ wood, fire, earth, metal, water }` の**カウント**
  - `meishiki.dayElement` = 日干の五行（その人の核）
  - `lucky.weakElement` / `lucky.supportElement` = **不足五行 / 補う五行**（用神に近い概念）
- 命名サービスも名前ごとに五行を計算済み（`wuxing`）。**両者は同じ5要素で表現されるため突合できる。**

### 1.4 なぜ六星占術は主役にしないか
六星占術エンジンは各人を**1つの星＝1つの五行**（土星人=土 等）に対応させるだけで、`gogyoCount` のような**バランス／不足プロファイルを出力しない**。診断の材料としては情報量が不足するため、**本連携の対象外**とする。

### Phase A — 五行計算を命名サービスに内蔵（自己完結）【先行】

四柱推命の計算ロジックを一元化する。方式は Phase B 着手時に確定（5章）。Phase A で切った**内部インターフェース（3.4）を固定**しておき、実装の中身だけ差し替えられるようにする。

## 3. データ互換性（最重要）

### 3.1 五行の対応表

| 概念 | 四柱推命エンジン | 命名サービス wuxing | 備考 |
|------|------------------|---------------------|------|
| 五行カウント | `gogyoCount.{wood,fire,earth,metal,water}` | `wuxing.{wood,fire,earth,metal,water}` | **キー名が完全一致**。そのまま突合可 |
| 支配的な五行 | （明示出力なし。max を取る） | `wuxing.dominant` | |
| 不足五行 | `lucky.weakElement`（最小カウントの要素、1つ） | `wuxing.lacking`（カウント0の要素、複数可） | **定義が異なる。3.3 で正規化ルールを規定** |
| 日干五行 | `meishiki.dayElement` | （なし） | 本人の核。コメント材料に活用 |

### 3.2 五行の算出方法の違い（重要な注意）
両者は「五行」という同じ言葉を使うが、**求め方が別系統**である。混同しないこと。

- **四柱推命**: 生年月日（＋時刻）→ 干支 → 各干支の五行を数える（陰陽五行の伝統的手法）。
- **命名サービス**: 画数の**一の位**から五行を決める（1,2=木 / 3,4=火 / 5,6=土 / 7,8=金 / 9,0=水。`wuxingOf()`）。

→ 連携で使うのは「**人の不足五行（四柱推命側）**」と「**名前が持つ五行（命名サービス側）**」の突合。**人側の五行は四柱推命の手法、名前側の五行は画数一の位の手法**で求める、という役割分担を厳守する。両者を同じ計算式に統一しようとしないこと。

### 3.3 不足五行の正規化ルール【Q2で確定：supportElement を含める】
四柱推命側の「不足五行」を、突合に使う**優先補完リスト `targetElements: Wuxing[]`（順序付き）**へ正規化する。

1. `gogyoCount` の中でカウント最小の五行を第1候補とする（＝ `weakElement` 相当）。
2. 同数最小が複数ある場合は全て候補に含める（`dayElement` は補完対象から除外＝自分の核は既に足りている扱い）。
3. さらに、日干を「生む」五行（`supportElement`、相生の親）を**第2候補として末尾に加える**。
4. 出力は重複除去した順序付き配列。例: `["water", "metal"]`（水が最も不足、次いで金を補うと良い）。

> `targetElements` を返す関数の**シグネチャを Phase A で固定**し、Phase B では実装だけ差し替える（2章の設計意図）。

### 3.4 内部インターフェース（Phase A/B で不変にする契約）
```ts
// api/_lib/bazi/types.ts
export interface BaziInput {
  birthDate: string;      // "YYYY-MM-DD"（必須。これだけで判定可能）
  birthTime?: string;     // "HH:mm"（任意。無ければ時柱なしで計算）
  birthPlace?: string;    // 都道府県コード等（任意。無ければ経度補正なし）
  timezone?: string;      // 既定 "Asia/Tokyo"
}
export interface WuxingBalance {
  counts: Record<Wuxing, number>;   // gogyoCount 相当
  dayElement: Wuxing;               // 日干の五行
  weakElement: Wuxing;              // 最も不足
  supportElement: Wuxing;           // 日干を生む五行
  targetElements: Wuxing[];         // 3.3 の正規化済み優先補完リスト
}
// Phase A: 内蔵計算 / Phase B: 四柱推命エンジン呼び出し に差し替え
export function calcWuxingBalance(input: BaziInput): WuxingBalance;
```

### 3.5 入力レベルと縮退動作【確定】

**生年月日だけで判定を完了できること。** 出生時刻・出生地はどちらも任意で、未入力でも処理を止めない。入力が増えるほど精度が上がる**積み上げ式**とする。

| レベル | 入力 | 計算内容 | 精度表示 |
|--------|------|----------|----------|
| **L1** | 生年月日のみ | **三柱（年・月・日）**で計算。時柱なし。時差補正なし | 「標準」 |
| **L2** | ＋出生時刻 | **四柱**（時柱を追加）。時差補正は明石基準のまま（経度補正なし） | 「詳しい」 |
| **L3** | ＋出生地（都道府県） | 四柱＋**経度補正＋均時差**（11.3.2） | 「最も詳しい」 |

**設計上の禁止事項**
- **未入力項目を勝手に補完しない。** 出生時刻未入力を「正午」、出生地未入力を「東京」等とみなすと、根拠のない結果を確定的に見せることになる。**時柱は生成せず、経度補正は適用しない**（＝その要素を計算から外す）。
- 縮退時は `WuxingBalance` に**どのレベルで計算したかを含め**、画面・テキスト出力で明示する（例: 「出生時刻の入力があると、より詳しく計算できます」）。
- L1でも `gogyoCount` は三柱分で成立し、身強／身弱の判定も**月令中心のため機能する**（11.3.1）。時柱がないぶん精度は落ちるが、判定自体は問題なく行える。

## 4. Phase A 詳細設計

### 4.1 モジュール配置
`api/_lib/bazi/`（先頭 `_` 配下なので Vercel の Function 化対象外）に、四柱推命の**純粋計算のみ**を TypeScript で移植する。

```
api/_lib/bazi/
  meishiki.ts   … buildMeishiki 相当（干支・gogyoCount 算出）。LLM・fetch・res 一切なし
  wuxing.ts     … calcWuxingBalance（3.4）／突合ロジック（4.3）
  types.ts      … BaziInput / WuxingBalance（Wuxing 型は fortune/sansai.ts を再利用）
  __tests__/    … 既知の生年月日で gogyoCount・weakElement・突合結果を固定検証
```

**移植元との対応（diagnose.js → bazi）**
- 移植する: `buildMeishiki`, `STEMS/BRANCHES/STEM_ELEMENT`, 蔵干テーブル `ZOUKAN_TABLE`, `SETSUIRI`, `pickMonthZoukan`, 年柱・月柱・日柱・（任意で時柱）算出, `gogyoCount` 集計, `calcLucky` の weak/support 算出部分。
- **移植しない**: `callAI` / 各プロバイダ呼び出し / プロンプト / バイオリズム / 日運 / 週間・月間データ / `handler`（HTTP層）。連携に不要。
- 五行キーは `wood/fire/earth/metal/water` で命名サービスの `Wuxing` 型と揃える（diagnose.js と同一キー）。

> 注意: diagnose.js の命式計算は簡易版（節入りテーブル `SETSUIRI` が概算、時差・真太陽時未考慮）。**占いの範囲として許容**し、命名サービスも同じ精度で移植する（両者の結果を一致させるため、あえて“正確化”しない）。Phase B で一本化されれば精度は自動的に揃う。

### 4.2 【重要】五行突合はスコア・ランクを変えない（独立表示）

Q1の回答を受けて設計を精査した結果、**五行突合を総合スコア／ランクに加算してはならない**。理由:

1. **共有URL（F-006）が壊れる**: 診断結果は `GET /api/diagnose?sei=...&mei=...&sex=...` で再現・共有する設計。一方、生年月日は**プライバシー上URLに載せない**（6章）。突合がスコアに影響すると、共有URLで開いた結果と本人が見た結果が食い違う。
2. **姓名判断の一貫性**: 「同じ名前・同じ性別なら同じ診断結果」という熊崎式の前提が崩れる。
3. **既存方針との整合**: 「診断結果の数値・ランクはロジック側で確定させる」という CLAUDE.md の原則を保つ。

→ **総合ランク・スコア・五格は生年月日に一切依存しない。** 五行突合は `wuxingMatch` という**独立した参考情報**として並記する。

### 4.3 五行ボーナスの判定【確定】

**呼称は「五行ボーナス」**とする。姓名判断の本体判定に対する「おまけ」であることを名称・表示の両方で伝える。**数値（0〜100）は出さず、3段階の言葉＋星**で表す（既存の「言葉が主役・点数は参考値」方針に合わせる）。

名前側の五行は既存の `wuxingOf()` で求める。**総格を主・三才（天/人/地）を従**として判定する（Q6の決定）。

```ts
export type BonusLevel = 3 | 2 | 1;   // 3=十分 / 2=やや / 1=補えていない

export interface WuxingBonus {
  targetElements: Wuxing[];   // その人が補いたい五行（3.3・四柱推命由来）
  soukakuElement: Wuxing;     // 総格の五行（名前の主たる五行）
  sansaiElements: Wuxing[];   // 三才の五行（天・人・地）
  matched: Wuxing[];          // 実際に補えている五行
  level: BonusLevel;          // 3段階
  stars: string;              // "★★★" / "★★☆" / "★☆☆"
  label: string;              // "十分に補えています" 等
  summary: string;            // 表示用の一文
  source: "shichu";           // 由来が四柱推命であることを明示（表示・出力で使用）
}
```

**判定ルール（設定値化）**

| 条件 | level | 星 | ラベル |
|------|-------|----|--------|
| 総格の五行が `targetElements[0]`（最も不足）に一致 | 3 | ★★★ | 十分に補えています |
| 総格が `targetElements[1]` 以降に一致、**または** 三才のいずれかが `targetElements[0]` に一致 | 2 | ★★☆ | やや補えています |
| いずれにも一致しない | 1 | ★☆☆ | 補えていません |

- level 1 のときは前向きな一言を添える（例「『水』を意識すると、バランスが整うかもしれません」）。
- `summary` 例: 「生年月日から見ると『水』が不足ぎみです。この名前は総格が水にあたるため、十分に補えています。」

> **【Q3の再定義・重要】** 当初の `WUXING_COMPLEMENT_WEIGHT`（0.25）は「提案スコアへの加点の重み」を想定した設定だった。しかしQ1（命名モード対象外）とQ4.2（総合スコアに加点しない）の決定により、**重み係数としての役割は消滅した**。「変更可能にする」というご要望の意図は、上表の**判定ルール自体を設定値として差し替え可能にする**ことで満たす（4.6）。数値の重みは持たない。

### 4.3.1 「四柱推命由来」であることの明示【確定】

ボーナスが姓名判断ではなく**四柱推命の考え方から算出されたもの**だと、ひと目で分かるようにする。実装時は以下すべてを満たすこと。

1. **ブロック見出しに明記**: 「四柱推命による五行ボーナス」
2. **由来バッジ／サブラベル**: 「生年月日から算出（四柱推命）」を見出し直下に小さく表示
3. **注釈文に明記**（4.4）: 四柱推命由来であること、姓名判断の結果には影響しないことの両方を書く
4. **テキスト出力（F-009/F-010）にも同じ見出しと由来を含める**（4.4・T-307）
5. **LLMコメント**に触れさせる場合も「四柱推命の考え方では」と前置きさせる（プロンプトで指示）

### 4.4 画面（S-001 入力 / S-002 結果）【Q1・Q4で確定】

### 4.5 API（`GET/POST /api/diagnose`）

- **POST** のリクエストに任意項目を追加: `birthDate?`, `birthTime?`, `timezone?`。

### 11.3.3 ④ 23時台の日柱の扱い【早子時説を採用】

**決定**: **早子時説**を採用。**23:00〜23:59 生まれは翌日の日柱**とする（現行の「繰り上げなし」から変更）。

- 実装: 出生時刻が 23:00 以降なら日柱計算用の日付を +1 日する。**時支は現行のまま**（`Math.floor(((h+1)%24)/2)` で23時→子と正しく出るため変更不要）。
- **適用条件**: 出生時刻（L2以上）が入力されている場合のみ。L1（生年月日のみ）では時刻が不明なため**適用しない**（3.5 の「未入力項目を勝手に補完しない」に従う）。
- 時差補正（⑤／11.3.2）を適用すると 23:00 境界をまたぐ場合がある。**補正後の時刻で判定する**こと。順序: 時差補正 → 早子時判定 → 日柱確定。
- 設定値 `BAZI_USE_EARLY_ZISHI`（既定 true）で切り替え可能にする。

> なお採用根拠は**四柱推命における早子時説**に置く。西洋占星術には23時で日付が変わる慣行はなく（太陽の黄経を出生時刻そのもので見る）、そちらを根拠にすると後で説明が食い違うため、ドキュメント上は早子時説を根拠として記載する。

### 11.3.4 ⑥ 蔵干の日数配分【現行を既定・算命学式も実装】

**本機能への影響**: 蔵干は (1) `gogyoCount`（月支の蔵干を+1）、(2) **身強／身弱の判定における通根**（⑦で用神論を採用したため）の2箇所に効く。例: 亥月生まれの前半7日間は現行「戊（土）」→算命学式「甲（木）」となり、土−1・木+1 で**不足五行の結論が変わり得る**。

**体系の整合に関する注意**: 出典に「水晶玉子式は時柱をとらない**三柱推命**であり、形としては算命学」とある通り、**算命学は本来 時柱を使わない体系**。本設計は⑦で用神論（子平・四柱推命の系譜）を採用し、L2/L3で時柱も使う。ここに算命学式の蔵干表だけを持ち込むと**体系が混ざる**（誤りというより「どの流派に立つか」が曖昧になる）。

### 11.3.1 ⑦「補うべき五行」の決め方【調査済み・用神論を採用】

**調査結果**: 現行実装と四柱推命の標準的な方法は、結論が異なるケースがある。

| | 現行実装 | 四柱推命の標準的な方法 |
|---|---|---|
| 考え方 | 五行を単純カウントし、**最も少ない五行を補う**（`weakElement`） | まず**日干の強弱（身強／身弱）**を判定し、それに応じて**用神**（吉となる五行）を決める |
| 身弱の場合 | （区別しない） | 日干を**支える**五行が用神＝**印星**（日干を生む五行）・**比劫**（日干と同じ五行） |
| 身強の場合 | （区別しない） | 日干を**漏らす／抑える**五行が用神＝**食傷・財・官** |

**つまり「少ないから補う」とは限らない。** 身強の人は、多い五行をさらに足すのではなく抑える方向が吉とされるため、現行方式とは逆の結論になる場合がある。

**身強／身弱の判定方法（流派共通の構造）**
1. **月令**（日干と月支の関係）が判定の中心。日干が月支に**通根**しているか（比肩・劫財等が月支にあるか）をまず見る
2. 月支以外の地支（年・日・時）への通根を加点
3. 日干以外の天干（年・月・時）による支えを加点
4. 合計点で 身強／中和／身弱 を判定

> **点数の刻み方・閾値は流派差がある**（「6点以上で最身強」「3〜14を身旺、0〜2を中和、−1〜−6を身弱」等、複数の方式が確認できた）。→ **閾値は設定値化する**（本プロジェクトで一貫している方式）。

**決定**: **用神論を採用する。** 理由は、本モジュールが最終的に四柱推命アプリのエンジンにもなるため、簡易方式のままでは本体側の精度を損なうため。

**現行コードからの流用**: `calcLucky` の `supportElement`（日干を生む五行）は**身弱時の用神＝印星と一致する**ため、半分は既存ロジックを活かせる。

**命名サービス側の表示への波及**: 「不足五行を補う」→「**あなたの用神（吉となる五行）を名前が含むか**」に変わる。3.3 の `targetElements` は「不足五行の優先リスト」から「**用神・喜神の優先リスト**」へ意味が変わる（型・関数シグネチャは不変）。表示文言も合わせて調整する（例:「あなたにとって吉となる五行は『水』です。この名前は水を含んでいます」）。四柱推命としてより正確な説明になる。

### 11.3.2 ⑤ 時差・真太陽時の実装方法（座標入力は不要）

**ユーザーに緯度経度を入力させる必要はない。** 必要なデータは2つで、どちらもオフライン完結・外部API不要。

1. **都道府県 → 代表経度のテーブル（47件）**: UIは都道府県セレクタ1つ。日本の四柱推命ソフトで一般的な方式。データ量は47行で、既存の文字マスタ3,168字に比べれば無視できる
2. **均時差**: **入力不要**。日付から計算式で求まる（年間±16分程度の周期変動）。近似式で十分

```
時刻補正 = (出生地の経度 − 135°) × 4分 + 均時差
```
135°＝明石（日本標準時の基準）。東京で約+19分、福岡で約−18分。

**適用条件**: 出生時刻（L2以上）が入力されている場合のみ意味を持つ。出生地未入力（L2）なら経度補正を適用せず処理を続行する（3.5）。

**注意**: 補正の結果、時刻が日境界や節入り境界をまたぐと日柱・月柱が変わり得る。境界処理をテストケースに含めること。

### 11.4 テスト設計上の注意【重要】

T-301 の完了条件を「`gogyoCount` が diagnose.js と一致」とすると、**移植ミスしか検出できない**。①②のように diagnose.js 自体が持つずれは、忠実に移植するほど同じようにずれるためテストを通過してしまう。

→ **diagnose.js との一致テストとは別に、万年暦（暦注計算等）の既知データとの突き合わせテストを必ず設ける**こと。③はこの方法で検証済み（11.1）。

### 11.5 四柱推命アプリとの整合（スコープへの波及）

①②を命名サービス側で修正すると、**四柱推命アプリと結果がずれる**（同じ生年月日で違う五行が出る）。これを避けるには修正を diagnose.js にも反映する必要があり、実質的に **Phase B（B-1 ロジック一元化）の前倒し着手**となる。

作業順序としてはむしろ自然:
1. `api/_lib/bazi/` を**最初から「①②修正込みの正しい実装」として作る**
2. 命名サービスで先に稼働させる
3. 四柱推命アプリ（diagnose.js）をこのモジュール利用に置き換える

「まず忠実に移植 → 後で改善」より二度手間が少ない。

## 9. リスク・制約

- **計算ロジック重複（Phase A）**: 四柱推命の修正が2箇所に必要な期間が生じる。Phase B で解消するまでの暫定と割り切る。
- **精度の限界**: 移植元が簡易命式（節入り概算・時差未考慮）。占いの範囲として許容。表記は断定を避ける。
- **五行手法の混同リスク**: 人＝伝統的五行、名前＝画数一の位、という役割分担を実装コメントとテストで明示し、取り違えを防ぐ。
- **共有URLとの非対称性**: 本人が見る結果には突合が出るが、共有URLでは出ない。**仕様として画面に明記**し、混乱を防ぐ（4.5）。
- **後方互換**: `POST /api/diagnose` は生年月日なしで完全に従来動作を維持すること（既存の共有URLを壊さない）。
- **別リポジトリ／別デプロイ**: エンジンは `gerupon-lgtm/test`、命名サービスは別。Phase B でリポジトリ構成の判断が必要。
