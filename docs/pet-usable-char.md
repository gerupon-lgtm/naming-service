# 「使いたい文字」再定義 設計書（F-003改訂）

ペット命名提案の「使いたい文字」を再定義する。

- ステータス: **実装済み**（2026-07-21・全テスト415件パス）
- バージョン: **`mvp-2.4.0`**（フロント変更＋出力文字種の追加を伴うため MINOR 繰上げ）
- 実装メモ: ローマ字表記はアルファベットに画数が無いため、**よみ（かな）で画数を数える**（`toItem` の `strokeSource`）。この点は実装で判明した必須事項。
- 関連: `命名サービス_要件定義書.md`（F-003）, `docs/api-design.md`, `docs/screens.md`, `docs/tasks.md`
- 実装対象: `api/_lib/pet/charNameLLM.ts`, `api/_lib/pet/suggest.ts`, `frontend/src/PetView.tsx`, `frontend/src/api.ts`

---

## 1. 背景（なぜ再定義するか）

v2.3.0 で「使いたい文字」を「かな=よみ照合／漢字=表記照合」に変えたが、次の問題があった。

- **漢字が実質使えない**: 判定は表記照合なのに、LLM生成プロンプトは「よみに含める」前提で書かれており、漢字を渡すと無意味な指示になり生成されなかった。
- **アルファベットが使えない**: 出力文字種が漢字・ひらがな・カタカナの3種のみで、ローマ字表記の名前を出せない。
- **かなが「希望のよみ」とほぼ同義**: よみ照合のため、かな指定は音の一部指定になり、別欄の「希望のよみ」と意味が重なる。

---

## 2. 再定義（確定仕様）

**「使いたい文字」= 単体の1文字を、名前の「表記（見た目の字）」に必ず含める。文字種は問わない。**

| 使いたい文字 | 表記に含まれる名前の例 |
|---|---|
| 空（漢字） | 空、あお空、空太 |
| も（ひらがな） | もも、もなか |
| モ（カタカナ） | モモ、モナカ |
| R（アルファベット） | R を含むローマ字表記の名前（例: Rui, Sora → R を含むもの） |

- **かなも「表記」照合**にする（v2.3.0 の「よみ照合」をやめる）。これで「希望のよみ」（音の指定）と明確に別物になる。
- **1文字限定**。入力欄を1文字に制限し、ラベルに「(1文字のみ)」と注記する。保険として、サーバーは複数文字が来ても**先頭1文字のみ**採用する。
- アルファベットの字は、名前が**ローマ字表記**でないと表記に現れない。したがってアルファベット指定を成立させるには**ローマ字出力**が必要（3章）。

---

## 3. 出力文字種に「ローマ字」を追加

- 出力文字種の選択肢を **漢字／ひらがな／カタカナ／ローマ字** の4種にする。
- **ローマ字はヘボン式**（パスポート標準。し=shi, つ=tsu, ち=chi, ふ=fu, じ=ji）。
- アルファベットの使いたい文字が指定されたら、**自動的にローマ字表記を対象**にする。
- **大文字小文字は同一視**する（「R」でも「r」でも可）。
- ヘボン式に現れない字（l・x・q・v 等）は候補が出にくい。**無理に作らせず、機械検証で弾く**（結果ゼロなら 5章のフォールバックに乗る）。「その字はローマ字名に出にくい」旨の案内を出すかは実装時に結果を見て判断。
- **矛盾チェック**（既存の漢字チェックと同様）: 使いたい文字がアルファベットなのに出力文字種でローマ字を外している場合は `INVALID_INPUT`。同様に、使いたい文字が漢字なのに漢字を外している場合等も従来どおり弾く。

---

## 4. プロンプトの3階層（重み付け）

条件を同じ強さで並べるとLLMがどれかを犠牲にするため、階層をつけて強弱をプロンプトに反映する。**先頭・強調ほど重く扱われる**ことを利用し、絶対条件を冒頭に置く。

| 階層 | 内容 | プロンプトでの扱い | 機械検証 |
|------|------|-------------------|----------|
| **Tier 1 絶対** | 使いたい文字が**表記に含まれる**／出力文字種を守る | 冒頭に「必ず・例外なく」 | **する**（満たさない出力は除外） |
| **Tier 2 強** | 性別・対象動物にふさわしい自然な名前 | 明示的に指定 | しない（スコアに反映） |
| **Tier 3 ソフト** | 雰囲気カテゴリ（かわいい・食べ物・和風） | 「**できれば**〜の雰囲気で」 | しない |

**Tier 1 は LLM 任せにしない。** LLM は条件を守らないことがあるため、生成後に「使いたい文字が表記に含まれるか」「出力文字種に合うか」「画数が引けるか」を必ずロジックで検証して弾く。

---

## 5. 件数とフォールバック

- **使いたい文字は絶対条件。** 満たす候補が少なくても妥協して埋めない。
- **1件でも出ればOK**。不足分はDB（`name_master`）から「表記に含む」条件で埋める（従前の仕組みを踏襲）。
- LLM 応答不可・条件を満たす出力ゼロなら、DB の「表記に含む」候補だけで動作する。

---

## 6. 「使いたい文字」＋「希望のよみ」を同時指定したとき（2段構え）

両方を Tier 1 にすると候補が激減し、そもそもよみが成立しないことが多い。次の2段で処理する。

1. **1回目**: 両立を要求する（例「『もなか』と読み、表記に『空』を含む名前」）。機械検証で「よみ一致 **かつ** 表記に使いたい文字」を通す。**1件以上残ればOK**、不足はDB埋め。
2. **両立ゼロなら2回目**: **よみを優先**して再検索する（「もなか」の名前）。使いたい文字は捨てる。（検索が2回になりレイテンシは増えるが許容）
3. **フォールバックのアラートを表示**する。

### アラートの受け渡し

- **サーバーが構造化情報を返す**（文言はフロントで組み立てる）。レスポンス例:
  ```json
  { "notice": { "kind": "reading_over_char", "droppedChar": "空", "reading": "もなか" } }
  ```
- フロントの表示文言（案）: 「使いたい文字『空』とよみ『もなか』を両立できる名前が見つからなかったため、よみを優先しました。」
- **保存・コピー出力にもこのアラートを含める**。

## 7. 画面（S-003 / S-004）

### S-003（入力）
- 「使いたい文字」欄を**1文字入力に制限**（`maxLength` と入力時トリム）。ラベルを「使いたい文字（任意・1文字のみ）」にする。
- 補足文言（案）: 「名前の表記にこの字を必ず入れます。漢字・ひらがな・カタカナ・アルファベットが使えます（アルファベットはローマ字表記の名前になります）。」
- 出力文字種チップに「**ローマ字**」を追加。

### S-004（結果）
- 使いたい文字由来の候補は由来チップ「使いたい文字から生成」（`source: "chars"`・既存）。
- **フォールバックのアラート**（6章）を結果上部に表示する。保存・コピー出力にも含める。

## 8. 決定事項の要約

| # | 論点 | 決定 |
|---|------|------|
| 1 | 使いたい文字の意味 | 単体1文字を**表記に含める**（文字種問わず）。かなも表記照合 |
| 2 | 文字数 | **1文字限定**。UIで制限＋ラベル注記。サーバーは先頭1文字のみ採用 |
| 3 | アルファベット | ローマ字出力を追加（ヘボン式）。大文字小文字は同一視。ヘボン式に無い字は無理に作らない |
| 4 | プロンプト | 3階層（絶対／強／ソフト）。Tier 1 は機械検証で担保 |
| 5 | 件数 | 使いたい文字は絶対。1件でもOK、不足はDB埋め。妥協しない |
| 6 | 使いたい文字＋よみ | 2段構え（両立→ダメならよみ優先）。フォールバック時はアラート表示 |
| 7 | アラート | サーバーが構造化情報を返し、フロントが文言化。保存・コピーにも含める |
