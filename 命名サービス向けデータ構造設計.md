# 命名サービス向けデータ構造設計 v11

## 目的

姓名判断エンジンを利用した命名サービスの基盤データベースを構築する。

対象：

- 人名
- 犬
- 猫
- 小動物
- 将来的な追加種別

本システムでは以下を軸とする。

- 画数による運勢評価
- 名前の響き
- 対象動物
- 性別
- テーマ（カテゴリ）

---

# 基本思想

命名サービスにおいて最も重要なエンティティは

```text
名前そのもの
```

である。

姓名判断結果は後から再計算できるように、

```text
結果
```

ではなく

```text
結果を導く元データ
```

を保持する。

## 熊崎式の部首補正について【v9追記／v10更新】

熊崎式は、一部の部首について新字体の画数ではなく伝統的な部首画数（旧字体基準）を用いる補正ルールを持つ（例: 氵→「水」の4画、扌→「手」の4画、艹→「艸」の6画等）。kanjiapi.dev（KANJIDIC2ベース）から取得できる画数はこの補正を反映しない現代の単純画数であるため、以下の運用とする。

- 補正が必要な代表的な部首・文字は、`character_master` のシードデータ作成時に手動で正しい画数に補正して投入する
- 未知文字をkanjiapi.devから自動取得・キャッシュする場合（F-007）、この補正は自動的には反映されない。稀な文字でのみ発生する限界として許容する【想定】（要件定義書 リスク欄参照）
- **【v10追記】半自動化の方針**: [Kanji alive](https://kanjialive.com/overview-jp/) のAPI（ビルド時のシードデータ生成スクリプトでのみ使用。実行時の外部連携ではない）から各文字の部首名・部首画数・位置情報を取得し、あらかじめ用意する「補正が必要な部首→補正値」の対応表（10〜20件程度を想定）と突き合わせて自動補正する。これにより、全漢字を目視確認する作業を「部首補正マッピング表の作成」という小さく閉じた作業に圧縮できる
- **制約**: Kanji aliveの対応漢字は1235字（常用漢字2136字の一部）にとどまるため、対応外の文字は引き続き手動確認が必要【要確認】
- **【v11追記】第二候補**: [MuzukanjiAPI](https://rapidapi.com/baqterya/api/muzukanjiapi)（RapidAPI経由、13,000字以上収録と説明されている）をKanji alive対応外の文字を埋める第二候補として検討する。ビルド時のみの利用のため実行時の可用性は問題にならないが、実際のレスポンス形式・無料枠のリクエスト上限は未確認のため、T-003着手時に検証してから採用可否を決める【要確認】

# 13. 会話ベースの決定事項ログ（v4追記）

設計レビューの過程で確認・決定した方針を記録する。データ構造そのものへの変更は最小限にとどめ、主に「各テーブルをどう使うか」の運用方針が中心。

## 13-1. 姓名診断と name_master の役割分離

- `name_master` は**提案（命名候補）専用**のテーブルとして使う。姓名診断（ユーザーが既存の姓名を入力して判定する機能）はこのテーブルを経由しない。
- 姓名診断は、入力された文字を都度 `character_master` で画数変換し、その場で計算する（候補データとして事前登録する必要はない）。
- したがって `fortune_result` / `fortune_detail` は、**提案用候補名（name_master に登録された名前）に対する事前計算結果**を保持するためのテーブルという位置づけになる。

## 13-2. 未知文字への対応（character_master 拡張）

- `character_master` は一般的な文字を網羅する方針。
- 網羅されていない文字（稀な異体字・辞書外の文字）が診断時に入力された場合、[kanjiapi.dev](https://kanjiapi.dev/) への問い合わせを検討する。
- API経由でのみ取得可能な文字は、**提案（候補名）側では使用しない**（候補として存在しないため）。姓名診断（都度計算）側でのみ発生し得る。
- 運用上の推奨: API から取得した文字は `character_master` に書き込み、以降は同じ文字を再度APIに問い合わせない（キャッシュ方式）。スキーマ変更は不要で、取得フローに「未知文字→API取得→character_masterへINSERT」を追加するだけで対応できる。
- kanjiapi.dev にも存在しない文字が来た場合の失敗時挙動（診断不可メッセージ等）は要件定義で決める。

## 13-3. ペット候補（苗字なし）の評価ロジック

- ペットの命名候補は苗字を考慮しないため、五格のうち**地格・総格のみを評価に使う簡易ロジック**とする。
- 姓が存在しない場合、定義上「地格＝総格」となり数値は一致するが、これは意図した仕様であり問題ない（地格＝総格を主要スコアとして扱う）。

## 13-4. 診断結果の保存方針

- 診断結果をDBに保存するテーブル（履歴等）は現時点では設けない。
- 無料サービスとして運用する前提のため、アカウント管理や保存用テーブルを増やさず、**結果をURLパラメータに埋め込んで共有可能にする**（ステートレスな再計算）方式を軸に検討する。
- 「過去の診断を一覧で見返す」機能は今回のスコープ外（必要になれば別途 `diagnosis_log` 等を要件定義で検討）。

## 13-5. 提案（命名候補）の絞り込みロジック

- 提案の算出は、当初案の「①score→②響き→③対象動物→④性別→⑤カテゴリ」の**順次フィルタ方式ではなく、各条件に重みを持たせた合成スコア方式**を採用する。
- 重みは条件ごとに設定可能とし、特に**性別の重みを重視**する。
- 重み付けはアプリケーション側の設定値（定数・configファイル等）で持たせる想定とし、DBスキーマの追加変更は不要。
- 「希望する響き（音節を含む）」の絞り込み方法（`first_sound` / `last_sound` / `mora_count` だけで十分か、`reading` への部分一致等が必要か）は追って検討する【要確認】。

# 14. 新機能: 文字指定・よみがな逆引き生成モード（v5追記・検討中）

以下はユーザーからの新規要望。詳細仕様は未確定のため、要望内容と懸念点を記録する。データ構造への反映は要件定義以降に行う。

## 14-1. 要望内容

- **使いたい文字の指定**: カンマ区切り・スペース区切り等で、使用したい文字を複数指定できるようにしたい。
- **よみがな逆引きモード**: 希望するよみがなを指定し、そのよみがなの中で良い画数になる文字（漢字）を選んで提案するモードが欲しい。
- **出力の文字種指定**: 結果を「漢字のみ」「ひらがな」「カタカナ」「アルファベット」「なんでもいい」から選べるようにしたい。

## 14-2. 懸念点・考慮漏れ（未確定・要件定義で詰める）

| # | 論点 | 内容 |
|---|------|------|
| 1 | 「使いたい文字」の意味 | **【確定・v5.2】must-include（結果に必ず含める文字）とする。** 候補プールの制限（その文字だけしか使えない）ではない |
| 1-1 | 使いたい文字の区切り方式 | **【確定・v8】半角・全角カンマ／半角・全角スペースいずれの区切りにも対応し、内部で配列に正規化する。** 複数指定時はAND条件（すべて含む） |
| 1-2 | 使いたい文字の位置指定 | **【確定・v8】不要（順不同でOK）。** 名前のどこかに含まれていればよい。位置指定（先頭固定等）は将来要望が出た場合に拡張候補とする |
| 1-3 | 使いたい文字と出力文字種指定の矛盾 | **【確定・v8】エラー表示とする。** 例: 出力＝ひらがなのみ、だが漢字を指定、のように矛盾する場合は自動調整せず、ユーザーに入力の修正を促す |
| 2 | 読み→漢字の逆引きデータ | **【確定・v6】`character_reading` のような読み用テーブルは新設しない。** [kanjiapi.dev](https://kanjiapi.dev/) の `GET /v1/reading/{reading}` が読み→漢字（main_kanji＝通常の音訓／name_kanji＝名乗り読み）を返すため、提案生成のたびにこのAPIをライブ呼び出しする方式とする。絞り込んだ漢字の画数は既存の `character_master`（stroke_countキャッシュ）から取得する |
| 3 | よみがなの分割の曖昧性 | **【確定・v8】拗音（きゃ・しゅ等）・長音（ー）・促音（っ）を1モーラとして正しく扱う簡易モーラ分割処理を用意する。** モーラ数に応じて考えられる区切り方（例:「はると」→ は+る+と／はる+と／は+ると／はると）を全パターン列挙し、各部分読みをkanjiapi.devのreadingエンドポイントで検索、最後まで組み立てられる組み合わせのみを候補として残す |
| 3-1 | 名前の文字数上限 | **【確定・v8】最大5文字までとする。** 列挙パターン数の発散を防ぐための上限 |
| 4 | 指定文字とよみがなの矛盾チェック | **【確定・v5.2】優先順位は「1. 使いたい文字（must-include） > 2. よみがな」。** 使いたい文字を確定要素とし、よみがなは残りの文字を選ぶ際の絞り込み条件として使う。両者が矛盾する場合はよみがな側を譲歩する（生成結果の読みがユーザー希望のよみがなと完全一致しないケースが起こり得る点はUI上の説明が必要）|
| 5 | ローマ字出力時の画数評価 | **【確定・v7】元の読み（かな）に変換し、そのかな表記の画数で代用評価する。** 既存の画数ベースロジック（character_master）をそのまま流用できる |
| 6 | 動的生成エンジンの必要性 | 既存の `name_master`（事前登録済み候補からの絞り込み）とは別に、読みから漢字を組み合わせて動的に候補を生成するエンジンが必要になる。組み合わせ爆発（同じ読みに該当する漢字が多数存在）と、不自然な組み合わせを大量生成してしまうリスクへの対策（生成数上限、実在感チェックの要否）を検討する |
| 7 | 対象範囲 | **【確定・v5.1】人間の命名は対象外。ペット命名（姓なし）のみを想定する。** 姓を含めた五格計算との両立は考慮不要 |
| 8 | スコープとコストのバランス | **【確定・v7】本機能一式（使いたい文字・よみがな逆引き・出力文字種指定）はMVPに含める。** 無料サービスの方針とは規模感が相反しうるため、6章（実現方式）の見直しは要件定義時に改めて検討する |
| 9 | 読み→漢字APIの障害時フォールバック | jisho.org（非公式・公式API不在）／Kanji alive（学習コンテンツ寄りで読み逆引き対応が未確認）／MuzukanjiAPI（RapidAPI経由、詳細仕様・料金未確認）の3候補を検討したが、**【確定・v7】MVPでは採用しない**。kanjiapi.dev単独＋簡易リトライ＋エラー表示にとどめ、実運用で障害が頻発する場合に改めてフォールバック実装を検討する |
